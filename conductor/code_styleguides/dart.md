# Dartコードスタイルガイド

このガイドは、公式のEffective Dartドキュメントから主要な推奨事項を要約したもので、スタイル、ドキュメンテーション、言語の利用法、API設計原則をカバーしています。これらのガイドラインに従うことで、一貫性があり、読みやすく、保守性の高いDartコードを促進します。

## 1. スタイル

### 1.1. 識別子

- **DO** 型、拡張機能、enum型には `UpperCamelCase` を使用してください。
- **DO** パッケージ、ディレクトリ、ソースファイルには `lowercase_with_underscores` を使用してください。
- **DO** インポートプレフィックスには `lowercase_with_underscores` を使用してください。
- **DO** その他の識別子（クラスメンバー、トップレベル定義、変数、パラメータ）には `lowerCamelCase` を使用してください。
- **PREFER** 定数名には `lowerCamelCase` を使用することを推奨します。
- **DO** 2文字より長い頭字語や略語は単語のように大文字化してください（例：`Http`、`Nasa`、`Uri`）。2文字の頭字語（例：`ID`、`TV`、`UI`）はすべて大文字のままにしてください。
- **PREFER** 無名関数およびローカル関数で未使用のコールバックパラメータにはワイルドカード（`_`）を使用することを推奨します。
- **DON'T** プライベートでない識別子に先頭のアンダースコアを使用しないでください。
- **DON'T** プレフィックス文字を使用しないでください（例：`kDefaultTimeout`）。
- **DON'T** `library` ディレクティブを使用してライブラリを明示的に命名しないでください。

### 1.2. 順序

- **DO** `dart:` インポートを他のインポートの前に配置してください。
- **DO** `package:` インポートを相対インポートの前に配置してください。
- **DO** エクスポートはすべてのインポートの後の別のセクションで指定してください。
- **DO** セクションをアルファベット順にソートしてください。

### 1.3. フォーマット

- **DO** `dart format` を使用してコードをフォーマットしてください。
- **CONSIDER** フォーマッタフレンドリーにするためにコードを変更することを検討してください（例：長い識別子を短くする、ネストした式を単純化する）。
- **PREFER** 1行を80文字以下にすることを推奨します。
- **DO** すべてのフロー制御文（`if`, `for`, `while`, `do`, `try`, `catch`, `finally`）に波括弧を使用してください。

## 2. ドキュメンテーション

### 2.1. コメント

- **DO** コメントは文のようにフォーマットしてください（最初の単語を大文字にし、ピリオドで終わる）。
- **DON'T** ドキュメンテーションにブロックコメント（`/* ... */`）を使用しないでください。通常のコメントには `//` を使用してください。

### 2.2. ドキュメントコメント

- **DO** `///` ドキュメントコメントを使用してメンバーと型を文書化してください。
- **PREFER** 公開APIのためにドキュメントコメントを書くことを推奨します。
- **CONSIDER** ライブラリレベルのドキュメントコメントを書くことを検討してください。
- **CONSIDER** プライベートAPIのためにドキュメントコメントを書くことを検討してください。
- **DO** ドキュメントコメントは単一文の要約で始めてください。
- **DO** ドキュメントコメントの最初の文を独自の段落に分けてください。
- **AVOID** 周囲のコンテキストとの冗長性を避けてください（例：クラスのドキュメントコメントでクラス名を繰り返さない）。
- **PREFER** 主な目的が副作用である関数やメソッドのコメントは、三人称の動詞で始めることを推奨します（例：「Connects to...」）。
- **PREFER** ブール値でない変数やプロパティのコメントは名詞句で始めることを推奨します（例：「The current day...」）。
- **PREFER** ブール値の変数やプロパティのコメントは、「Whether」に続いて名詞句または動名詞句で始めることを推奨します（例：「Whether the modal is...」）。
- **PREFER** 値を返すことが主な目的である関数やメソッドには、名詞句または非命令形の動詞句を推奨します。
- **DON'T** プロパティのゲッターとセッターの両方にドキュメントを書かないでください。
- **PREFER** ライブラリまたは型のコメントは名詞句で始めることを推奨します。
- **CONSIDER** ドキュメントコメントにトリプルバッククォートを使用してコードサンプルを含めることを検討してください。
- **DO** ドキュメントコメント内でスコープ内の識別子を参照するには角括弧（`[]`）を使用してください（例：`[StateError]`、`[anotherMethod()]`、`[Duration.inDays]`、`[Point.new]`）。
- **DO** パラメータ、戻り値、例外を説明するために散文を使用してください。
- **DO** メタデータアノテーションの前にドキュメントコメントを配置してください。

### 2.3. Markdown

- **AVOID** Markdownを過度に使用しないでください。
- **AVOID** フォーマットにHTMLを使用しないでください。
- **PREFER** コードブロックにはバッククォートフェンス（```）を推奨します。

### 2.4. 書き方

- **PREFER** 簡潔さを推奨します。
- **AVOID** 明白でない限り、略語や頭字語を使用しないでください。
- **PREFER** メンバーのインスタンスを参照するには、「the」の代わりに「this」を使用することを推奨します。

## 3. 使用法

### 3.1. ライブラリ

- **DO** `part of` ディレクティブには文字列を使用してください。
- **DON'T** 他のパッケージの `src` ディレクトリ内にあるライブラリをインポートしないでください。
- **DON'T** インポートパスが `lib` の内外に及ぶことを許可しないでください。
- **PREFER** `lib` の境界を越えない場合は、相対インポートパスを推奨します。

### 3.2. Null安全

- **DON'T** 変数を明示的に `null` に初期化しないでください。
- **DON'T** `null` の明示的なデフォルト値を使用しないでください。
- **DON'T** 等価演算で `true` または `false` を使用しないでください（例：`if (nonNullableBool == true)`）。
- **AVOID** 初期化されているかどうかを確認する必要がある場合は `late` 変数を避けてください。null許容型を推奨します。
- **CONSIDER** null許容型を使用するために、型昇格またはnullチェックパターンを検討してください。

### 3.3. 文字列

- **DO** 文字列リテラルを連結するために隣接する文字列を使用してください。
- **PREFER** 文字列と値を構成するために補間（`$variable`、`${expression}`）を使用することを推奨します。
- **AVOID** 必要ない場合は補間で波括弧を使用しないでください（例：`'$name'` の代わりに `'${name}'`）。

### 3.4. コレクション

- **DO** 可能な場合はコレクションリテラル（`[]`, `{}`, `<type>{}`）を使用してください。
- **DON'T** コレクションが空かどうかを確認するために `.length` を使用しないでください。`.isEmpty` または `.isNotEmpty` を使用してください。
- **AVOID** 関数リテラルで `Iterable.forEach()` を使用しないでください。`for-in` ループを推奨します。
- **DON'T** 結果の型を変更する意図がない限り `List.from()` を使用しないでください。`.toList()` を推奨します。
- **DO** 型でコレクションをフィルタリングするために `whereType()` を使用してください。
- **AVOID** 近くの操作（`List<T>.from()` や `map<T>()` など）で代用できる場合は `cast()` を使用しないでください。

### 3.5. 関数

- **DO** 関数を名前にバインドするには関数宣言を使用してください。
- **DON'T** ティアオフで済む場合にラムダを作成しないでください（例：`list.forEach(print)` の代わりに `list.forEach((e) => print(e))`）。

### 3.6. 変数

- **DO** ローカル変数の `var` と `final` について一貫したルールに従ってください（再代入されない場合は `final`、再代入される場合は `var`、またはすべてのローカルに `var`）。
- **AVOID** 計算できるものを保存しないでください（例：`radius` がある場合に `area` を保存しない）。

### 3.7. メンバー

- **DON'T** 不必要にフィールドをゲッターとセッターでラップしないでください。
- **PREFER** 読み取り専用プロパティを作成するために `final` フィールドを使用することを推奨します。
- **CONSIDER** 単純なメンバー（ゲッター、セッター、単一式のメソッド）には `=>` を使用することを検討してください。
- **DON'T** 名前付きコンストラクタへのリダイレクトまたはシャドウイングを避ける場合を除き、`this.` を使用しないでください。
- **DO** 可能な場合は宣言時にフィールドを初期化してください。

### 3.8. コンストラクタ

- **DO** 可能な場合は初期化仮引数（`this.field`）を使用してください。
- **DON'T** コンストラクタ初期化子リストで済む場合に `late` を使用しないでください。
- **DO** 空のコンストラクタ本体には `{}` の代わりに `;` を使用してください。
- **DON'T** `new` を使用しないでください。
- **DON'T** 定数コンテキストで冗長に `const` を使用しないでください。

### 3.9. エラー処理

- **AVOID** `on` 句のない `catch` 句を避けてください。
- **DON'T** `on` 句のない `catch` 句からのエラーを破棄しないでください。
- **DO** プログラムエラーに対してのみ `Error` を実装するオブジェクトをスローしてください。
- **DON'T** `Error` またはそれを実装する型を明示的にキャッチしないでください。
- **DO** 元のスタックトレースを保持するために、キャッチした例外を再スローするには `rethrow` を使用してください。

### 3.10. 非同期

- **PREFER** 生の `Future` を使用するよりも `async`/`await` を推奨します。
- **DON'T** 有用な効果がない場合に `async` を使用しないでください。
- **CONSIDER** ストリームを変換するために高階メソッドを使用することを検討してください。
- **AVOID** `Completer` を直接使用しないでください。

## 4. API設計

### 4.1. 名前

- **DO** 用語を一貫して使用してください。
- **AVOID** 省略されていない用語よりも一般的でない限り、略語を避けてください。
- **PREFER** 最も説明的な名詞を最後に置くことを推奨します（例：`pageCount`）。
- **CONSIDER** APIを使用する際にコードが文のように読めるようにすることを検討してください。
- **PREFER** ブール値でないプロパティや変数には名詞句を推奨します。
- **PREFER** ブール値のプロパティや変数には非命令形の動詞句を推奨します（例：`isEnabled`、`canClose`）。
- **CONSIDER** 名前付きブール値パラメータの動詞を省略することを検討してください（例：`growable: true`）。
- **PREFER** ブール値のプロパティや変数には「肯定的な」名前を推奨します（例：`isDisconnected` ではなく `isConnected`）。
- **PREFER** 主な目的が副作用である関数やメソッドには命令形の動詞句を推奨します（例：`list.add()`、`window.refresh()`）。
- **PREFER** 値を返すことが主な目的である関数やメソッドには、名詞句または非命令形の動詞句を推奨します（例：`list.elementAt(3)`）。
- **CONSIDER** 実行する作業に注意を引きたい場合は、関数やメソッドに命令形の動詞句を検討してください（例：`database.downloadData()`）。
- **AVOID** メソッド名の先頭に `get` を付けないでください。
- **PREFER** オブジェクトの状態を新しいオブジェクトにコピーする場合は、メソッド名を `to___()` とすることを推奨します（例：`toList()`）。
- **PREFER** 元のオブジェクトに裏打ちされた異なる表現を返す場合は、メソッド名を `as___()` とすることを推奨します（例：`asMap()`）。
- **AVOID** 関数やメソッド名でパラメータを説明しないでください。
- **DO** 型パラメータに名前を付ける際は、既存のニーモニック規則に従ってください（例：要素には `E`、マップのキー/値には `K`、`V`、一般的な型には `T`、`S`、`U`）。

### 4.2. ライブラリ

- **PREFER** 宣言をプライベートにすること（`_`）を推奨します。
- **CONSIDER** 論理的にまとまっている場合は、同じライブラリで複数のクラスを宣言することを検討してください。

### 4.3. クラスとミックスイン

- **AVOID** 単純な関数（`typedef`）で済む場合に、メンバーが1つの抽象クラスを定義しないでください。
- **AVOID** 静的メンバーのみを含むクラスを定義しないでください。トップレベルの関数/変数またはライブラリを推奨します。
- **AVOID** サブクラス化を意図していないクラスを拡張しないでください。
- **DO** クラス修飾子（例：`final`、`interface`、`sealed`）を使用して、クラスが拡張可能かどうかを制御してください。
- **AVOID** インターフェースであることを意図していないクラスを実装しないでください。
- **DO** クラス修飾子を使用して、クラスがインターフェースになれるかどうかを制御してください。
- **PREFER** `mixin class` よりも純粋な mixin または純粋なクラスを定義することを推奨します。

### 4.4. コンストラクタ

- **CONSIDER** クラスがサポートしている場合（すべてのフィールドが `final` でコンストラクタで初期化される）、コンストラクタを `const` にすることを検討してください。

### 4.5. メンバー

- **PREFER** フィールドとトップレベル変数を `final` にすることを推奨します。
- **DO** 概念的にプロパティにアクセスする操作にはゲッターを使用してください（引数なし、結果を返す、ユーザーに見える副作用なし、べき等）。
- **DO** 概念的にプロパティを変更する操作にはセッターを使用してください（単一引数、結果なし、状態を変更する、べき等）。
- **DON'T** 対応するゲッターなしにセッターを定義しないでください。
- **AVOID** オーバーロードを偽装するためにランタイム型テストを使用しないでください。
- **AVOID** 初期化子なしの public な `late final` フィールドを避けてください。
- **AVOID** null許容の `Future`、`Stream`、コレクション型を返さないでください。空のコンテナまたはnull許容型の非null許容futureを推奨します。
- **AVOID** メソッドチェーンを有効にするためだけにメソッドから `this` を返さないでください。メソッドカスケードを推奨します。

### 4.6. 型

- **DO** 初期化子のない変数に型注釈を付けてください。
- **DO** 型が自明でない場合は、フィールドとトップレベル変数に型注釈を付けてください。
- **DON'T** 初期化されたローカル変数に冗長な型注釈を付けないでください。
- **DO** 関数宣言に戻り値の型注釈を付けてください。
- **DO** 関数宣言にパラメータの型注釈を付けてください。
- **DON'T** 関数式で推論されたパラメータ型に注釈を付けないでください。
- **DON'T** 初期化仮引数に型注釈を付けないでください。
- **DO** 推論されないジェネリック呼び出しに型引数を記述してください。
- **DON'T** 推論されるジェネリック呼び出しに型引数を記述しないでください。
- **AVOID** 不完全なジェネリック型を記述しないでください。
- **DO** 推論が静かに失敗する代わりに `dynamic` で注釈を付けてください。
- **PREFER** 関数型注釈のシグネチャを推奨します。
- **DON'T** セッターに戻り値の型を指定しないでください。
- **DON'T** レガシーな `typedef` 構文を使用しないでください。
- **PREFER** `typedef` よりもインライン関数型を推奨します。
- **PREFER** パラメータに関数型構文を使用することを推奨します。
- **AVOID** 静的チェックを無効にしたい場合を除き、`dynamic` を使用しないでください。
- **DO** 値を生成しない非同期メンバーの戻り値の型として `Future<void>` を使用してください。
- **AVOID** 戻り値の型として `FutureOr<T>` を使用しないでください。

### 4.7. パラメータ

- **AVOID** 位置指定のブール値パラメータを避けてください。
- **AVOID** ユーザーが前のパラメータを省略したい場合がある場合は、オプションの位置指定パラメータを避けてください。
- **AVOID** 特殊な「引数なし」の値を受け入れる必須パラメータを避けてください。
- **DO** 範囲を受け入れるには、包括的な開始パラメータと排他的な終了パラメータを使用してください。

### 4.8. 等価性

- **DO** `==` をオーバーライドする場合は `hashCode` をオーバーライドしてください。
- **DO** `==` 演算子が等価性の数学的ルール（反射的、対称的、推移的、一貫性）に従うようにしてください。
- **AVOID** 可変クラスに対してカスタムの等価性を定義しないでください。
- **DON'T** `==` のパラメータをnull許容にしないでください。
